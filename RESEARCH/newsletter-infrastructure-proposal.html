02 The Big Pieces

Six core components that you need to build. Each can be developed and tested independently, then wired together.

1

Data Ingestion Engine

A set of fetcher modules — one per API source. Each normalizes data into a standard JSON schema. A scheduler (cron or GitHub Actions) triggers collection on a cadence.

2

AI Writer Core

The Claude-powered writing engine. Reads ingested data, applies a personality/style system prompt, generates each newsletter section, and outputs structured Markdown.

3

Git Workflow Manager

Automates the git lifecycle: create branch, commit draft, open PR, assign reviewers. Merge-to-main triggers the send pipeline. All history is auditable.

4

Review & Editing Layer

PR-based review where editors can comment, suggest changes, or push commits. Optional: a lightweight web UI that renders the Markdown preview alongside the PR.

5

Template & Rendering Engine

Converts the Markdown draft into a responsive HTML email using templates (MJML or custom). Also generates a web-archive version and RSS entry.

6

Delivery Pipeline

Integrates with an ESP (SendGrid, Resend, Mailchimp). Handles subscriber lists, send scheduling, bounce management, and analytics tracking.

03 The AI Writer — Deep Dive

This is the heart of the system. The writer's quality depends on how well you architect its prompt stack, personality management, and tool use. Here's how to build it right.

3.1 — Prompt Architecture (The Stack)

The writer isn't one prompt — it's a layered system of prompts that compose together. Each layer is a file in the repo, version-controlled, reviewable, and tweakable by any team member.

prompts/ ├── personality.md # Core voice, tone, persona definition ├── style-guide.md # Writing rules: length, formatting, do/don't ├── audience-profile.md # Who the reader is, what they care about ├── sections/ │ ├── intro.md # Prompt for writing the opening section │ ├── main-stories.md # Prompt for the featured stories block │ ├── quick-hits.md # Prompt for short news roundup │ ├── analysis.md # Prompt for the opinion/analysis piece │ └── sign-off.md # Prompt for the closing section ├── examples/ │ ├── good-edition.md # A gold-standard edition for few-shot reference │ └── bad-patterns.md # Anti-patterns to avoid └── assembly.md # Master prompt that orchestrates section assembly
Key Insight: Every prompt file lives in git. When someone wants to tweak the newsletter's voice or add a new section, they make a PR against the prompts directory — just like code. This means personality changes are versioned, reviewed, and reversible.

3.2 — Personality Management

The personality file is the most important prompt. It defines who the writer "is." This should be a rich, specific document — not a vague instruction like "be friendly." Think of it as a character bible.

# Example: prompts/personality.md

## Voice
You are a sharp, well-read industry insider writing a weekly newsletter
called "The Signal." You sound like a smart friend who happens to know
everything about [your industry]. You're opinionated but fair, concise
but never shallow.

## Tone Spectrum
- Default tone: Conversational-professional (think: Stratechery meets Morning Brew)
- When covering serious news: Direct and measured, no jokes
- When covering fun/weird stories: Witty, playful, occasional dry humor
- When giving analysis: Confident but show your reasoning

## Hard Rules
- Never use: "dive in", "buckle up", "game-changer", "at the end of the day"
- Never start with "Welcome to this week's edition"
- Always attribute claims to sources
- Use "we" sparingly — this newsletter has a singular author voice
- Maximum one exclamation mark per edition

## Signature Moves
- Open with a strong, unexpected angle on the week's biggest story
- Use a recurring sign-off phrase: "[Your catchphrase here]"
- Include one contrarian take per edition — flag it as "Hot Take:"

3.3 — Claude API Integration Design

The writer uses the Claude API with tool use for a multi-step generation workflow. Here's the recommended architecture:

# writer/core.py — Simplified orchestration

import anthropic
import json

client = anthropic.Anthropic()

def generate_edition(ingested_data: dict, edition_date: str) -> str:
    # 1. Load the prompt stack from repo files
    personality = read_file("prompts/personality.md")
    style_guide = read_file("prompts/style-guide.md")
    audience    = read_file("prompts/audience-profile.md")
    example     = read_file("prompts/examples/good-edition.md")

    # 2. Build the system prompt — personality + style + audience
    system_prompt = f"""
{personality}

---
STYLE GUIDE:
{style_guide}

---
AUDIENCE:
{audience}

---
REFERENCE EDITION (for tone/structure, not content):
{example}
"""

    # 3. Build the user message with ingested data
    user_message = f"""
Write the newsletter edition for {edition_date}.

Here is today's ingested data:
{json.dumps(ingested_data, indent=2)}

Follow the section structure defined in the style guide.
Use the tools provided to cite sources and fact-check claims.
"""

    # 4. Call Claude with tool use for citations and fact-checking
    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=8192,
        thinking={"type": "adaptive"},
        system=system_prompt,
        tools=WRITER_TOOLS,   # cite_source, format_section, etc.
        messages=[{"role": "user", "content": user_message}]
    )

    return extract_markdown(response)

3.4 — Writer Tools

Define these as Claude API tools so the writer can self-structure its output and maintain quality:

ToolPurposeWhy It Matters
cite_source Attaches a URL/source to a claim Forces proper attribution; generates footnotes automatically
format_section Wraps output in section metadata (title, type, word count) Ensures consistent structure; enables per-section rendering
rank_stories Scores stories by relevance/impact before writing Claude decides what's worth covering, not just what's available
check_tone Self-evaluates a paragraph against the personality spec Catches voice drift before it hits the draft
add_commentary Flags where the writer is adding opinion vs. reporting Transparency for reviewers; readers know what's editorial

3.5 — Quality Control Pipeline

Before the draft is committed, run automated quality checks:

Q1

Self-Review Pass

A second Claude call reads the draft against the style guide and personality file, then scores it and suggests revisions. Use a cheaper model (Haiku) for cost efficiency.

Q2

Source Verification

Script checks all cited URLs are reachable and haven't 404'd. Flags any claims without citations. Optionally uses Claude to cross-check facts against source content.

Q3

Structural Validation

Linting pass — checks word counts per section, verifies all required sections exist, validates Markdown syntax, checks for banned phrases from personality file.

04 Git-Centric Collaboration

Git is the single source of truth. Every edition, every prompt change, every reviewer comment — all tracked in the repository. This is what makes the system multi-editor without chaos.

4.1 — Repository Structure

newsletter-repo/ ├── prompts/ # All prompt files (personality, style, sections) ├── templates/ # HTML email templates (MJML or Handlebars) ├── data/ │ ├── sources.yaml # API source definitions and configs │ └── raw/ # Ingested data snapshots (gitignored or LFS) ├── editions/ │ ├── 2026-02-23/ │ │ ├── data.json # Ingested data for this edition │ │ ├── draft.md # AI-generated Markdown draft │ │ ├── final.md # Post-review final version │ │ └── metadata.json # Subject line, send time, stats │ └── 2026-02-16/ │ └── ... ├── writer/ # AI writer Python/TS code ├── ingestion/ # Data fetcher modules ├── delivery/ # Send pipeline code ├── .github/workflows/ # CI/CD: generate, test, send ├── config.yaml # Global config (schedule, recipients, API keys ref) └── README.md

4.2 — Branch & PR Strategy

main ──────────────────────●───────────────────────────●──────── (sends) ↑ ↑ edition/2026-02-23 ────●───● │ ↑ review commits │ │ │ prompt/update-voice ──────────────────────────●──────────● ↑ team approves
  • main — The "send" branch. Merging an edition here triggers delivery. Always in a sendable state.
  • edition/YYYY-MM-DD — Auto-created by the writer. Contains the draft. PR is opened for review. Reviewers push commits to fix issues.
  • prompt/* — For prompt/personality changes. Requires team review before merging because it affects all future editions.

4.3 — GitHub Actions Automation

WorkflowTriggerWhat It Does
generate.yml Scheduled (cron) or manual dispatch Runs ingestion, calls AI writer, commits draft, opens PR
quality-check.yml PR opened/updated on edition/* Runs linting, source verification, spell check, renders preview
send.yml Push to main (edition merged) Renders HTML, sends via ESP, updates web archive
prompt-test.yml PR on prompts/* Generates a test edition with the new prompts, posts preview to PR
Multi-editor flow: Anyone on the team can review an edition PR — fix typos, rewrite a section, add a story. They just push commits to the edition branch. The final merge to main is the "send approval." Require at least one approval before merge for safety.

05 Data Ingestion Layer

A modular, pluggable system for pulling data from any source and normalizing it for the writer.

5.1 — Source Configuration

# data/sources.yaml
sources:
  - name: hacker_news
    type: api
    url: https://hacker-news.firebaseio.com/v0
    fetcher: ingestion/fetchers/hackernews.py
    schedule: "daily"
    max_items: 30

  - name: twitter_industry
    type: api
    fetcher: ingestion/fetchers/twitter.py
    config:
      lists: ["ai-researchers", "tech-leaders"]
    schedule: "daily"

  - name: company_blog
    type: rss
    url: https://example.com/blog/feed.xml
    fetcher: ingestion/fetchers/rss_generic.py

  - name: internal_analytics
    type: custom
    fetcher: ingestion/fetchers/analytics.py
    secrets_ref: ANALYTICS_API_KEY

08 Multi-Editor Roles & Permissions

Who can do what — mapped to GitHub's native permission model.

RoleCan DoGitHub Role
Editor-in-Chief Approve & merge edition PRs, modify prompts, change config Admin / Maintainer
Section Editor Push edits to edition branches, review PRs, comment Write
Contributor Suggest edits via PR comments, review but not merge Triage

Branch protection rules: Set main to require at least 1 approving review before merge. This ensures no edition ships without a human sign-off. You can also require specific reviewers for prompt changes.
